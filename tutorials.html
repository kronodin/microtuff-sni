SNI Bypass Configuration
Server Name Indication tunneling for Caribbean networks

SNI (Server Name Indication) reveals the hostname your client connects to via TLS. ISPs use this to block specific services. SNI bypass tricks the ISP into seeing a different, whitelisted hostname while routing to your actual destination.

How It Works

Your TLS handshake sends a SNI field that ISPs inspect. By connecting through a proxy that sets the SNI to a CDN or whitelisted domain (like whatsapp.com or a Cloudflare endpoint), the ISP allows the connection while your proxy forwards to the real target.

3-7 Day SSH Tunnel Setup
Persistent SSH tunnels with auto-reconnect

Create persistent SSH tunnels that auto-reconnect and last multiple days using systemd service management.

Requirements

  • VPS with public IP (DigitalOcean $6/mo, Vultr free tier, or Oracle Always Free)
  • OpenSSH server + stunnel or sslh for TLS wrapping
  • HTTP Injector on Android, Shadowrocket on iOS, or OpenSSH client on desktop

Step 1: Generate SSH Keys

ssh-keygen -t ed25519 -f ~/.ssh/microtuff_key -N ""
ssh-copy-id -p 443 user@your-vps-ip

Step 2: Create Systemd Service

File: /etc/systemd/system/ssh-tunnel.service

[Unit]
Description=Microtuff SSH Tunnel
After=network.target

[Service]
Type=simple
User=tunneluser
ExecStart=/usr/bin/ssh -o ServerAliveInterval=60 -o ServerAliveCountMax=3 -o ExitOnForwardFailure=yes -N -R 8080:localhost:80 -p 443 user@your-vps-server.com -i /home/tunneluser/.ssh/microtuff_key
Restart=always
RestartSec=30
RestartLimitBurst=10

[Install]
WantedBy=multi-user.target

Step 3: Enable and Run

sudo systemctl daemon-reload
sudo systemctl enable ssh-tunnel
sudo systemctl start ssh-tunnel
sudo systemctl status ssh-tunnel

Step 4: Configure Client (HTTP Injector)

[Connection]
Type: SSH
Host: YOUR_VPS_IP
Port: 443
Username: user
Private Key: microtuff_key

[Proxy]
Type: HTTP
Host: 127.0.0.1
Port: 8080

[Payload]
GET / HTTP/1.1
Host: target-site.com
Connection: keep-alive
💡 Duration: With ServerAliveInterval=60, the tunnel stays alive indefinitely. The systemd RestartSec=30 auto-reconnects every 30 seconds if it drops. This easily lasts 3-7 days without intervention. For longer duration, rotate keys weekly.

WebSocket Tunnel Configuration
Persistent connections with upgrade headers

WebSocket upgrades headers carry metadata for persistent connections. The upgrade request can route traffic through encrypted tunnels while appearing as normal HTTPS.

Sample WebSocket Request

GET / HTTP/1.1
Host: secure-websocket.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Version: 13
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Protocol: chat, superchat
Origin: https://www.example.com

Key Configuration Points

  • Host: Your proxy server (the one ISPs see)
  • SNI in TLS: Can differ from the HTTP Host header
  • Origin: Proxied through — hides your actual origin
  • Sec-WebSocket-Key: Random 16-byte base64 per connection
💡 Tip: Works best with Shadowrocket or HTTP Injector WebSocket plugin. Set ws-host and ws-origin headers to match your target.

USB-over-WAN Remote Repair Setup
Share USB devices remotely for repair diagnostics

Allow clients to share USB devices remotely to your repair technician dashboard (Repair Connect).

Client Instructions (Zero Install)

  1. Open Repair Connect page in Chrome or Edge
  2. Enable WebUSB mode in chrome://flags (if needed)
  3. Click "Connect Device" button
  4. Plug USB device into computer
  5. Browser will detect the USB device — click "Share" to allow remote access
  6. Issuer is notified — device is now accessible to repair technician via WebUSB API

Supported USB Devices

  • USB flash drives and external hard drives
  • Android phones (ADB mode / file transfer mode)
  • iPhones (limited — requires trusted host pairing)
  • Printers and scanners
  • Keyboards, mice, game controllers
⚠️ Note: Client must keep browser tab open. Connection ends when they close the page. WebUSB requires HTTPS (GitHub Pages provides this automatically).

Advanced SNI Bypass Methods
Deep packet inspection evasion techniques

1. ClientHello Fragmentation

Split the TLS ClientHello handshake across multiple TCP packets. DPI systems that inspect single packets cannot extract the SNI field. Implemented in tools like uTLS (used by utls-client) and GOST.

2. Domain Fronting

Set SNI to a high-reputation CDN domain (e.g., cdn.cloudflare.com) while sending a different HTTP Host header for your actual target. Works on any CDN that accepts arbitrary host headers. Cloudflare and Google Cloud CDN are known compatible.

3. ECH (Encrypted Client Hello)

TLS 1.3 extension that encrypts the entire ClientHello including SNI. The ISP sees only encrypted bytes where the hostname should be. Native support in Firefox (network.dns.echcfg) and Chrome (Encrypted ClientHello). This is the future of SNI privacy.

4. H3 QUIC Protocol

HTTP/3 uses QUIC protocol over UDP 443 instead of TCP 443. Most DPI systems only inspect TCP traffic. Switching to QUIC-based connections bypasses the TCP TLS inspection entirely. Use with curl --http3 or v2ray VLESS+QUIC.

5. Session Resumption

TLS session tickets allow reconnection without a full handshake — including skipping the SNI exchange. Enable server-side: ssl_session_tickets on; (Nginx) and client-side session caching. Tools like ss-local support session ticket reuse.

6. Multi-SNI Rotation

Rotate through dozens of SNI domains in rapid succession. Even if one SNI triggers a block, the next connection uses a different one. Effective against reactive DPI systems that block specific SNIs after detection.

💡 Pro Tip: Combine methods for maximum bypass capability: SNI bypass + WebSocket upgrade + ECH encryption + QUIC transport = full-spectrum network bypass.
